## Findings

### BUG 1 — Root .bat files have LF-only line endings (Windows .bat requirement: CRLF)

**Files affected:** `hermes-chat.bat`, `hermes-deepseek.bat`, `hermes-deepseek-pro.bat`

**Severity:** High — .bat files with LF-only endings are silently treated as a single-line command by cmd.exe on some Windows versions. The `@echo off` and `title` lines do not take effect; users see a verbose execution trace in the terminal.

**Root cause:** Files were likely created or last edited on a Unix/Git Bash environment without explicit CRLF write, or committed without `core.autocrlf=true`. The `generate-launchers.cjs` generator correctly writes CRLF (`lines.join("\r\n") + "\r\n"`) — the generated files in `launchers/out/` are CRLF-correct. Only the hand-committed root .bat files were broken.

**Fix applied:** Rewrote all three files with explicit CRLF line endings using `printf '...\r\n'`. Verified with Python byte-level check: CRLF=3, LF-only=0 for each.

### BUG 2 — `install.sh` Step 4: `\L` in sed replacement is broken in Git Bash

**File:** `sync/install.sh`, lines 153 and 158 (original)

**Severity:** High — this is the primary Desktop path detection code path. Both the cmd.exe branch (line 153) and the `$USERPROFILE` fallback (line 158) used the same broken sed pattern. When sed fails, it exits with an error and produces an empty string. `DESKTOP_DIR` becomes empty, the `[[ -d "$DESKTOP_DIR" ]]` guard fails, and the installer prints "Could not detect Desktop path" — skipping both `hermes-chat.bat` copy and the `Hermes Launchers/` folder creation.

**Root cause:** `\L` is a GNU sed replacement-side modifier (lowercase). It is documented in GNU sed 4.0+ but does NOT function in the sed bundled with Git for Windows 4.9 when used in the `s|||` replacement side — verified to produce `sed: unknown option to s` and return an empty string on the test machine.

**Additionally:** The `cmd.exe /c "echo %USERPROFILE%"` call (line 151) does not produce just the USERPROFILE value. It outputs the full Windows banner (2 lines) + a blank line + the CWD prompt, with no echo value visible. This is a known behavior when cmd.exe is invoked from Git Bash. The `$USERPROFILE` env var (set by Windows, inherited by Git Bash) is the reliable source.

**Fix applied:** Replaced the entire `DESKTOP_DIR` detection block with:
1. `cygpath -u "$USERPROFILE"` (primary — Git for Windows built-in, always available, produces clean lowercase `/c/Users/...` form)
2. `echo "$USERPROFILE" | sed 's|[\\]|/|g; s|^\([A-Za-z]\):|/\1|'` fallback (uses character class `[\\]` to match backslash reliably, removes the broken `\L`, accepts uppercase drive letter — valid on Windows case-insensitive FS)

The cmd.exe branch was removed entirely (unreliable as shown above).

**Verified:** Running the fixed logic against `USERPROFILE=C:\Users\J` produces `/c/Users/J/Desktop`, which exists and lists files correctly.
